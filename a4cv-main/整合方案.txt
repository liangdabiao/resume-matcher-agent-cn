## 现状梳理

| 项 | 简历优化器（前端 + 后端） | a4cv-main 编辑器 |
|---|---|---|
| 技术栈 | Next.js 14 + FastAPI | 纯静态单页 `index.html` + `vendor/` |
| 数据流 | 简历/岗位 → `/improve`（HR Judge Prompt）→ 返回 `analysis_result` 整篇 Markdown 报告 | 接受 Markdown → `parseMD()` → 渲染到 A4 画布 |
| 关键发现 | Prompt 的 Step 4 明确要求 LLM 在 \`\`\`md 代码块里输出"修改后简历范本"，但前端 dashboard 直接把整篇报告 HTML 渲染了，**没人专门抽这个代码块** | 有 `loadMD(md)` 和 `normalizeImportedMarkdown()` 两个入口，但没有 URL/sessionStorage 接收外部投递的逻辑（`URLSearchParams` 全文 0 匹配） |
| a4cv 内的 AI 优化 | — | 已经内置 DeepSeek/OpenAI 兼容接口的「✦ AI 优化」按钮 |

**核心结论**：两边都没有为"对接"留接口，需要做的工作主要是 ① 抽出"修改后简历"代码块 ② 给 a4cv 加启动钩子接收外部投递 ③ 解决同源问题。

### 为什么选 sessionStorage 而不是 URL 传参

| 维度 | URL `?md=...` | sessionStorage |
|---|---|---|
| 简历 markdown 通常 2-8KB | URL 通常 ≤2KB 会被截断，浏览器差异大 | 5MB 同源限制，足够 |
| 同源要求 | 不需要 | 需要同源 |
| 实现复杂度 | 需 LZ-String 压缩 | 几行代码 |
| 用户体验 | URL 又丑又长 | 干净 |

**sessionStorage 要求同源**，所以关键配套步骤是把 a4cv-main 部署成 Next.js 静态资源：`apps/frontend/public/a4cv/`，让两者共用 `localhost:3000`。

```python
@resume_router.get("/improved-markdown")
async def get_improved_markdown(
    resume_id: str, job_id: str,
    db: AsyncSession = Depends(get_db_session),
):
    """
    从已生成的 analysis_result 中抽取 \`\`\`md 代码块；
    没有则用 Resume.content 拼成最小可用 Markdown。
    """
    import re
    # 1) 读最近的 score_improvement 执行结果（需要给 ProcessedResume 加字段或临时重跑）
    #    简化：重跑一次 /improve 拿 analysis_result
    score_service = ScoreImprovementService(db=db)
    result = await score_service.run(resume_id=resume_id, job_id=job_id)
    text = result.get("analysis_result", "")
    m = re.search(r"```(?:md|markdown)?\s*([\s\S]+?)\s*```", text)
    if m and "# " in m.group(1):
        return {"markdown": m.group(1).strip()}
    # 2) fallback: 直接读原始简历
    resume = await db.get(Resume, UUID(resume_id))
    return {"markdown": (resume.content or "").strip()}
```

## 风险与边界

1. **同源是关键约束**：必须把 a4cv 放进 Next.js `public/` 或反代到同域。**不可**用 Netlify 跨域托管。
2. **sessionStorage 容量**：5MB 远大于一份简历 markdown，足够。
3. **数据隐私**：sessionStorage 在 tab 关闭后自动清除，符合"临时投递"语义，且不污染 a4cv 自身的 `localStorage`。
4. **a4cv 内"AI 优化"按钮的 API Key**：仍由用户在 a4cv 端自己填写（沿用它现有的 UI 和 localStorage），不污染我们后端账户体系。
5. **Prompt 改进**：当前 `hr_judge.py` 已要求 LLM 输出代码块，但格式不一定稳；可考虑在 Step 4 后追加一行强化指令保证 \`\`\`md...\`\`\` 包裹。
6. **a4cv 的模块识别关键字**（[parseMD 第 2046 行](file:///d:/resume-upup/resume-matcher-agent-cn/a4cv-main/index.html#L2046)）：自我评价 / 工作经历 / 项目经历 / 教育背景 / 技能标签 / 证书与荣誉 —— 我们后端 prompt 输出的代码块应**对齐这套命名**，否则部分模块会进 `mixed` 类型。后端 prompt 已经在自我评价/关键成果/工作经历/项目经历/教育背景/技能标签/证书与荣誉之间，**基本兼容**，仅"关键成果"会被识别为 `list` 类型（可正常显示）。

| 方案 | 不推荐原因 |
|---|---|
| 纯 URL 传 `?md=...` | URL 长度限制 + 中文 encode 难看 + 用户手动改 URL 风险 |

```tsx
// apps/frontend/app/(default)/editor/page.tsx
'use client';
import Script from 'next/script';
import { useEffect } from 'react';
import { useSearchParams } from 'next/navigation';
import { useResumePreview } from '@/components/common/resume_previewer_context';

export default function EditorPage() {
  const params = useSearchParams();
  const { improvedData } = useResumePreview();

  // a4cv 状态桥：监听外部数据并调用 a4cv 的 loadMD
  useEffect(() => {
    if (!improvedData?.data?.analysis_result) return;
    fetch(`${API_URL}/api/v1/resume/improved-markdown?resume_id=${improvedData.data.resume_id}&job_id=${improvedData.data.job_id}`)
      .then(r => r.json())
      .then(({ markdown }) => {
        if (typeof window.loadMD === 'function') {
          window.loadMD(markdown);
        } else {
          // a4cv 还没初始化好，挂到全局回调
          window.__pendingMD = markdown;
        }
      });
  }, [improvedData]);

  return (
    <>
      {/* 1) 顺序加载 vendor（必须保持原 index.html 顺序） */}
      <Script src="/a4cv-vendor/marked.min.js" strategy="afterInteractive" />
      <Script src="/a4cv-vendor/Sortable.min.js" strategy="afterInteractive" />
      <Script src="/a4cv-vendor/html2canvas.min.js" strategy="afterInteractive" />
      <Script src="/a4cv-vendor/jspdf.umd.min.js" strategy="afterInteractive" />
      <Script src="/a4cv-vendor/mammoth.browser.min.js" strategy="afterInteractive" />
      
      {/* 2) 注入 a4cv 的 CSS（用 CSS Module 隔离） */}
      <link rel="stylesheet" href="/a4cv-runtime.css" />
      
      {/* 3) 渲染 a4cv 的 body 模板 */}
      <div dangerouslySetInnerHTML={{ __html: A4CV_BODY_HTML }} />
      
      {/* 4) 注入 a4cv 的逻辑脚本（包裹 IIFE） */}
      <Script id="a4cv-runtime" strategy="afterInteractive">{`
        (function(){
          ${A4CV_SCRIPT_BODY}
          // 暴露入口给 React 调用
          window.loadMD = loadMD;
          window.serializeMarkdown = serializeMarkdown;
          window.showToast = showToast;
          // 消费 React 投递的待加载 md
          if (window.__pendingMD) { loadMD(window.__pendingMD); window.__pendingMD = null; }
        })();
      `}</Script>
    </>
  );
}
```

#### Step 3：CSS 隔离（关键）
a4cv 有 ~1300 行内联 CSS，会污染全局。**两种策略二选一**：
- **A. CSS-in-JS 注入**：把 a4cv 的 CSS 放到 `apps/frontend/app/(default)/editor/a4cv-runtime.css`，Next.js 默认会做 CSS 局部 scope
- **B. Shadow DOM**：把整个 a4cv 渲染到 `<div ref={hostRef}>` + `attachShadow`，最彻底但 a4cv 内部用了 `document.fonts` 等全局 API，要兼容
- **推荐 A**：成本低、风险小、a4cv 的类名都比较独特（`r-` 前缀、`preset-` 前缀、`tbtn` 等），冲突概率低

#### Step 4：状态桥（React ↔ a4cv）
```tsx
// components/editor/a4cv-bridge.ts
declare global {
  interface Window {
    loadMD: (md: string) => void;
    serializeMarkdown: () => string;
    showToast: (msg: string) => void;
    __pendingMD?: string;
  }
}
```
- dashboard 的"打开编辑器"按钮：
  ```tsx

  const handleOpenEditor = () => {
    sessionStorage.setItem('pendingResumeId', data.resume_id);

  ```
- editor 页面通过 `useSearchParams` + `sessionStorage` 拿到 ids，调后端接口拿 markdown，再调 `window.loadMD()`

#### Step 5：UI 一致性（按需）
- 用 Tailwind 重做顶部工具栏（保留 a4cv 的"导入文档""AI 优化""导出"按钮）
- 替换品牌名（Resume Studio → 简历优化器）
- 移除 a4cv 的"公众号二维码""载入示例"等私货
- 把 a4cv 的 `localStorage` 键名加前缀 `a4cv_` 避免冲突

### 与上一版方案 A 的对比

| 维度 | 方案 A（sessionStorage 中转） | **方案 P5（搬入 Next.js）** |
|---|---|---|
| 同源问题 | 需要复制到 `public/` | 天然同源 |
| 状态共享 | sessionStorage 间接 | **直接 React Context** |
| 资源加载 | a4cv 自己 fetch | 走 Next.js 的 `<Script>` 优化 |
| CSS 作用域 | 不需要处理 | 需做 CSS 隔离 |
| 后续可演进 | 仍然是 a4cv 独立 UI | **可逐步替换为 React + Tailwind** |
| 与 a4cv 上游同步 | 简单（直接覆盖） | 中等（要 merge 改动） |
| 实现工作量 | 0.5 人天 | 2-3 人天 |
| 长期价值 | 工具型集成 | **产品化整合** |

## 风险与缓解

| 风险 | 缓解措施 |
|---|---|
| a4cv 的 CSS 与 Tailwind 冲突 | a4cv 自身类名有 `r-`、`tbtn`、`preset-` 等独特前缀；放独立 CSS 文件 + 编辑器页用 `BackgroundContainer` 不包内层 |
| a4cv 的全局 `window` 污染 | 用 IIFE 包裹，暴露明确接口（`window.loadMD` 等） |
| a4cv 用了 `window.module/exports/define` 切换 | a4cv 自己已处理（在 index.html 第 12-14 行），照搬即可 |
| `localStorage` 键名冲突 | 改用 `a4cv_resumeAiSettings`、`a4cv_resumeProfiles.v1` 等前缀 |
| Next.js 的 strict mode 双调用 `useEffect` | a4cv 内部 `state` 是模块单例，React 18 双调用可能导致重复初始化 → 用 `useRef` 锁 |
| 字体（Google Fonts）跨域 | 保留原有 `<link>` 即可 |
| `app/(default)/layout.tsx` 的 `BackgroundContainer` 与 a4cv 的暗色顶栏冲突 | editor 页用自己的 layout（用 Next.js 的 Route Groups 隔离） |
| 移动端 | a4cv 是桌面端设计，复用其响应式 |

---

## 推荐的最终方案：**P5 渐进式搬入**

**为什么是这个：**
1. **契合用户原话"搬过去"**：内化而非外挂
2. **风险可控**：先搬后改，每步都可回滚
3. **数据流最干净**：React Context 直接对接 a4cv 的 `loadMD`，无需中转
4. **未来可演进**：CSS/UI 逐步替换为 React + Tailwind，体验向系统原生靠拢
5. **保留 a4cv 上游升级路径**：搬入后 a4cv 升级只需重新拷贝 vendor + 重新生成 page 模板
